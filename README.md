# The Marchetti Project OS V0717

A new, production-grade operating system for creating, generating, quality-checking, managing, publishing, and scaling the AI influencer **Kaia Marchetti**.

This is a completely new greenfield project.

No files, rules, approvals, decisions, content, code, canon documents, prompts, schemas, assets, or workflows from any previous project are part of this project.

---

## Project Purpose

The Marchetti Project OS V0717 is being built from scratch to create a reliable daily operating system for Kaia Marchetti.

The system will support:

* Consistent AI influencer generation.
* Daily image and video production.
* Identity protection.
* Structured prompt creation.
* Quality-control reviews.
* Owner approval.
* Content packaging.
* Publishing operations.
* Performance tracking.
* Revenue growth.

The purpose is to replace disconnected prompts, scattered images, and inconsistent workflows with one controlled production system.

## Core Principles

### 1. Build from verified information

The system must not invent missing information.

Unknown information must remain unknown until the owner makes a decision.

Example:

```json
{
  "value": null,
  "status": "PENDING_OWNER_DECISION"
}
```

---

### 2. Kaia’s identity must be controlled

Kaia’s identity will be defined through project-specific canon files and approved visual references created for V0717.

The system must not:

* Invent identity facts.
* Change approved features.
* Treat generated images as automatic canon.
* Replace approved visual references.
* Fill unresolved fields without approval.
* Allow providers to redefine Kaia.

---

### 3. Every generated asset must be traceable

Every approved asset should eventually connect to:

```text
Daily Brief
→ Identity Snapshot
→ Reference Manifest
→ Generation Job
→ Compiled Prompt
→ Provider Settings
→ Raw Output
→ Quality-Control Record
→ Owner Approval
→ Platform Export
→ Publication Record
→ Analytics Record
```

---

### 4. Owner approval remains required

During the controlled production stage, the system may prepare work but must not independently:

* Change Kaia’s identity.
* Approve generated assets.
* Publish content.
* Accept partnerships.
* Make unsupported claims.
* Spend money.
* Delete source assets.
* Replace approved files.

Final authority remains with Tara Nelson.

---

### 5. The system must support daily use

The finished workflow must be simple enough to operate every day.

The daily process should eventually require the owner to:

1. Choose a business objective.
2. Complete or approve a daily brief.
3. Review generated assets.
4. Approve, reject, or request corrections.
5. Approve platform content.
6. Review performance results.

The system should handle repetitive technical work without removing owner control.

```text

4. Approve the daily brief.
5. Load the approved Kaia identity.

7. Select approved visual references.
8. Create the generation job.
9. Compile provider-specific prompts.
10. Generate image or video assets.
11. Run identity and technical quality control.
12. Correct failed assets.
13. Review final assets.
14. Approve or reject each asset.
15. Prepare platform-specific versions.
16. Publish approved content.
17. Record performance.
18. Use verified results to improve future content.
```
